# SAR Auction Project Architecture Primer

## 1. Why This Document Exists

This document is a beginner-friendly map of how this MEAN project is organized.

It explains:

- how frontend and backend are separated
- what each main folder/file is for
- where dependencies are installed
- where build/run scripts are defined
- which files students should edit to implement features

Detailed day-to-day coding workflow is documented in:

- `Backend/DEVELOPMENT.md`
- `Frontend/DEVELOPMENT.md`

## 2. Big Picture: How the App Works

This project has two main codebases:

1. Frontend (Angular) in `Frontend/`
2. Backend (Express + Node.js + MongoDB + Socket.IO) in `Backend/`

At runtime:

1. Browser requests the app.
2. Backend serves the built Angular app from `dist/auction-sar`.
3. Frontend calls backend REST APIs under `/api/*`.
4. Backend reads/writes MongoDB.
5. Socket.IO sends real-time updates (bids, user/activity events).

## 3. Repository Layout (What Lives Where)

Top-level folders and key files:

- `Frontend/`: Angular source code (components, services, models, routing)
- `Backend/`: Express/Node TypeScript source code (routes, controllers, models, middleware)
- `dist/auction-sar/`: built frontend output (generated by Angular build)
- `Backend/dist/`: built backend output (generated by TypeScript build)
- `angular.json`: Angular workspace build/serve/test configuration
- `package.json` (root): frontend scripts and frontend dependencies
- `Backend/package.json`: backend scripts and backend dependencies
- `tsconfig*.json` (root): frontend TypeScript config
- `Backend/tsconfig.json`: backend TypeScript config

## 4. Frontend vs Backend Separation

### Frontend responsibilities (`Frontend/`)

- Render UI screens (signin/register/auction/items)
- Validate forms and user interactions
- Call backend APIs through Angular services
- Subscribe to socket events for real-time updates

### Backend responsibilities (`Backend/`)

- Expose API endpoints under `/api`
- Authenticate and authorize requests
- Apply business rules for auction logic
- Persist data in MongoDB via Mongoose
- Broadcast real-time events with Socket.IO

## 5. Frontend Folder

Inside `Frontend/app/`:

- `core/`
- Shared app-wide building blocks
- `core/services/`: HTTP/socket/auth services used across features
- `core/models/`: TypeScript data models/interfaces
- `core/guards/`: route protection (auth guard)

- `features/`
- Feature modules and feature components students interact with most
- `features/auth/`: signin/register screens
- `features/auction/`: auction dashboard behavior
- `features/items/`: item insertion/management behavior

- `material/`
- Angular Material module imports

- `shared/`
- Shared reusable UI/module pieces

Also important:

- `Frontend/main.ts`: Angular app bootstrap
- `Frontend/index.html`: base HTML page
- `Frontend/environments/`: environment files

## 6. Backend Folder

Inside `Backend/src/`:

- `server.ts`
- Backend entry point; server setup, middleware setup, HTTPS/HTTP behavior

- `routes/`
- API route declarations and endpoint mapping

- `controllers/`
- Request handlers (parse request, call logic, send response)

- `services/`
- Reusable business logic and socket service logic

- `models/`
- Mongoose schemas/models for MongoDB collections

- `middlewares/`
- Cross-cutting concerns (auth, error handling)

- `config/`
- Configuration and database connection setup

## 7. Dependency and node_modules Management

This repository has two dependency contexts:

1. Root dependencies (frontend toolchain and Angular libs)

- Defined in root `package.json`
- Installed into root `node_modules/`

2. Backend dependencies

- Defined in `Backend/package.json`
- Installed into `Backend/node_modules/`

What students should remember:

- If you change root `package.json`, run install at root.
- If you change `Backend/package.json`, run install in backend context.
- Keep frontend/backend dependency changes separated.

## 8. Where Build/Run/Test Scripts Are Defined

### Root scripts (frontend)

Defined in root `package.json`:

- `npm start` -> Angular dev server (`ng serve`)
- `npm run build` -> Angular build output to `dist/auction-sar`
- `npm test` -> Angular tests (Karma)

Angular build/serve behavior is configured in `angular.json`:

- `sourceRoot`: `Frontend`
- build output path: `dist/auction-sar`
- test entry: `Frontend/test.ts`

### Backend scripts

Defined in `Backend/package.json`:

- `npm --prefix Backend run dev` -> backend watch mode (`tsx watch src/server.ts`)
- `npm --prefix Backend run build` -> compile backend TypeScript to `Backend/dist`
- `npm --prefix Backend run start` -> run compiled backend (`node dist/server.js`)
- `npm --prefix Backend run lint` -> backend lint checks

## 9. Build Outputs and Runtime Mode

There are two `dist` folders and both are expected:

1. `dist/auction-sar`

- Frontend production build artifacts

2. `Backend/dist`

- Compiled backend JavaScript artifacts

In this project setup, backend can serve the built frontend from `dist/auction-sar`.

## 10. Files Students Will Usually Edit

Most student feature work should happen in these areas:

Frontend:

- `Frontend/app/features/**/*`
- `Frontend/app/core/services/**/*`
- `Frontend/app/core/models/**/*`
- `Frontend/app/core/guards/**/*`

Backend:

- `Backend/src/routes/api.routes.ts`
- `Backend/src/controllers/**/*`
- `Backend/src/services/**/*`
- `Backend/src/models/**/*`
- `Backend/src/middlewares/**/*`

Students should rarely edit these unless necessary:

- build config files (`angular.json`, `tsconfig*`)
- server bootstrap (`Backend/src/server.ts`)
- dependency manifests (`package.json` files)

## 11. Request and Data Flow (Mental Model)

When adding a new feature, think in this order:

1. Frontend UI captures input.
2. Frontend service sends API request.
3. Backend route maps request to controller.
4. Controller/service applies logic.
5. Model persists/reads data.
6. Backend responds with JSON.
7. Frontend updates UI state.
8. Optional: socket event notifies other connected clients.

## 12. Practical Guidance for New MEAN Students

1. Start by locating the feature in `Frontend/app/features/`.
2. Trace which frontend service calls the backend.
3. Follow the endpoint in `Backend/src/routes/api.routes.ts`.
4. Implement behavior in controller/service/model layers.
5. Validate with browser flow plus backend logs.
6. Keep changes focused to one feature path at a time.

This approach prevents random edits and helps you learn full-stack flow step by step.
